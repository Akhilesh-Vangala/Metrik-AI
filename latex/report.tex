\begin{abstract}
This report describes Metrik~AI, a Python pipeline that predicts hourly building energy consumption, flags anomalies in the predictions, and ranks buildings for operational audit. We work with the ASHRAE Great Energy Predictor~III dataset---20.2 million hourly meter readings from 1,449 buildings across 16 sites. The pipeline runs out-of-core with chunked I/O, builds 38 leakage-safe features, and trains a LightGBM model that cuts RMSE by 82\% relative to a per-meter mean baseline. A modified Z-score detector flags 44,363 anomalous hours (2.2\% of validation), and a composite scorer produces a ranked audit list of 964 building--meter pairs. We benchmark several acceleration techniques from the course---NumPy vectorization, Cython, Numba, CuPy, and multiprocessing---and report measured speedups ranging from 1.5$\times$ (parallel training) to 36.5$\times$ (vectorized scoring) on real data.
\end{abstract}

\section{Introduction}
% ============================================================================

Buildings are responsible for roughly 40\% of total U.S.\ energy consumption, and a large portion of that energy is wasted~\cite{miller2020}. The root cause is straightforward: most commercial buildings still run heating, cooling, and lighting on fixed schedules rather than responding to actual demand. If a building operator had an accurate hourly forecast of energy use, they could pre-cool before a heatwave, catch a malfunctioning chiller before the monthly bill arrives, or figure out which of their 200 buildings to retrofit first.

That is the problem we tackle here. Metrik~AI takes hourly meter readings, weather, and building metadata, and produces three outputs: (1)~per-meter consumption forecasts, (2)~anomaly flags on the residuals, and (3)~a ranked audit list that tells an operator where to look first. This three-part structure mirrors the Measurement \& Verification (M\&V) workflow that energy services companies actually use in practice~\cite{molina2017}.

On the Python side, the project gave us a real excuse to use most of the techniques from DS-GA~1019 on a non-trivial dataset: chunked I/O so we don't blow up memory, vectorized feature engineering, compiled kernels in Cython and Numba for the hot loops, GPU code via CuPy and Numba CUDA, process- and thread-based parallelism, a PySpark pipeline for the distributed case, and \texttt{scipy.optimize} for hyperparameter search. None of these were bolted on after the fact---each one addresses a specific bottleneck we hit while building the pipeline.

\section{Data}
% ============================================================================

We use the ASHRAE Great Energy Predictor~III dataset from Kaggle~\cite{miller2020}. It comes from the Building Data Genome Project~2 and contains real operational meter readings, not synthetic data. The training set has 20.2\,M rows: hourly readings from 1,449 non-residential buildings (universities, offices, hotels, warehouses, etc.) spread across 16 sites in North America and Europe. Each row is a \texttt{(building\_id, meter, timestamp, meter\_reading)} tuple, where meter type is one of electricity (0), chilled water (1), steam (2), or hot water (3). Two auxiliary tables give us building metadata (square footage, year built, primary use) and site-level hourly weather.

The data has some well-known quirks. Site~0 reports electricity in kBTU instead of kWh---we convert at load time. Several meters have long zero-reading streaks that indicate outages, not actual zero consumption; we flag streaks $\geq$48\,h and replace them with NaN. Weather columns are 10--40\% missing depending on the site, so we impute per-site with forward-fill then backward-fill. Outliers are capped at the 99.9th percentile per building-meter group. Table~\ref{tab:data_quality} gives a snapshot of the per-site quality after cleaning.

The whole training CSV is about 1.3\,GB, which is too large to load in one shot on a laptop without pushing memory past 4\,GB once we merge in metadata and weather. So we read it in chunks of 2\,M rows using \texttt{pandas.read\_csv(chunksize=...)} and wrap the reader in a Python generator that yields cleaned, merged chunks one at a time. In dev mode we limit chunks with \texttt{itertools.islice}. This keeps peak memory under 4\,GB for the entire pipeline.

\section{Methodology}
% ============================================================================

\subsection{Features}

We build 38 features, split into six groups:

\begin{itemize}[noitemsep,leftmargin=*]
\item \textbf{Time (10):} hour, day-of-week, month, weekend flag, holiday flag (looked up via the \texttt{holidays} library with \texttt{@lru\_cache} to avoid redundant date parsing), day-of-year, and sine/cosine encodings for hour and month so the model sees cyclical continuity.
\item \textbf{Lags (3):} the meter reading from 24\,h ago, 168\,h ago (same time last week), and the 24\,h difference.
\item \textbf{Rolling stats (8):} mean, std, min, and max over the previous 24\,h and 168\,h windows. These are shifted by one step so the window never includes the current timestep---this is the main leakage-prevention mechanism, and we have unit tests that verify it.
\item \textbf{Building (3):} log of square footage, building age, and primary-use category code.
\item \textbf{Weather (8):} air temperature, dew temperature, their difference, estimated relative humidity, squared temperature (for non-linear HVAC response), wind chill, cloud coverage, wind speed.
\item \textbf{Interactions (4):} hour$\times$weekend, temperature$\times$hour, sqft$\times$temperature, and the ratio of current reading to its rolling mean.
\end{itemize}

Everything is stored as \texttt{float32} to halve memory relative to the default \texttt{float64}. In the codebase, the final feature list is assembled from these six sub-lists using \texttt{itertools.chain} so we can easily add or remove groups.

We also keep a deliberately slow ``naive'' version of the feature builder that uses \texttt{iterrows} and Python-level loops. Its only purpose is to serve as a baseline for profiling---comparing the vectorized version against it is how we measure the speedup from NumPy/pandas vectorization.

\subsection{Models}

We do a time-based train/validation split: the last 3 months of data go to validation, the rest to training. No shuffling---this preserves temporal order and is the right way to evaluate a forecasting model.

Three models:

\begin{enumerate}[noitemsep,leftmargin=*]
\item \textbf{Per-meter mean.} Compute the average reading for each (building, meter) pair on the training set and predict that constant for every hour. This is the simplest possible baseline.
\item \textbf{Lag-24h.} Use yesterday's reading at the same hour as the forecast. Surprisingly strong for buildings with stable daily patterns.
\item \textbf{LightGBM}~\cite{ke2017}. 63 leaves, learning rate 0.05, bagging fraction 0.8, early stopping after 50 rounds of no improvement on the validation set. Categorical features (primary use, site, meter type) are handled natively by LightGBM without one-hot encoding. We also wrote an \texttt{optimize\_learning\_rate} function that wraps \texttt{scipy.optimize.minimize\_scalar} (bounded Brent's method) to search for the best learning rate in $[0.005, 0.3]$ by minimizing validation RMSE.
\end{enumerate}

\subsection{Anomaly Detection}

After training, we compute residuals (actual $-$ predicted) on the validation set and score them per building-meter group using the modified Z-score:
\begin{equation}
M_i = \frac{0.6745\,(x_i - \tilde{x})}{\text{MAD}}, \quad \text{MAD} = \text{median}(|x_i - \tilde{x}|)
\end{equation}
where $\tilde{x}$ is the group median. The 0.6745 constant makes it consistent with a standard normal distribution under Gaussian assumptions. Hours with $|M_i| > 3.5$ are flagged.

We then cluster consecutive flagged hours into ``streaks.'' A building with a 72-hour anomaly streak is a much bigger concern than one with three isolated spikes, so the streak length feeds into the final priority scoring.

\subsection{Decision Support}

The audit list ranks building-meter pairs by a composite score that combines anomaly rate, mean severity, total excess energy (positive residuals summed), and longest streak. Operators can sort this list and start inspecting from the top. In our run, this produces 964 entries.

\section{Optimization and Acceleration}
% ============================================================================

The pipeline has several computationally heavy parts: scoring millions of residuals, building rolling-window features, and training per-site models. We used profiling (\texttt{cProfile} for CPU time, \texttt{tracemalloc} for memory) to find the actual bottlenecks, then applied the right tool for each one.

\subsection{Vectorization}

The single biggest win. Replacing a Python \texttt{for} loop over 1\,M residuals with \texttt{np.median} and array arithmetic drops runtime from 0.66\,s to 0.018\,s---a 36.5$\times$ speedup. No compilation, no dependencies, just NumPy. Similarly, building 38 features on 50K rows takes 0.73\,s with \texttt{iterrows} and 0.14\,s with vectorized pandas (5.3$\times$). The lesson is clear: before reaching for Cython or Numba, try vectorizing first.

\subsection{Cython}

We wrote three functions in \texttt{cython\_kernels.pyx}: modified Z-score, rolling mean, and rolling standard deviation. The code uses \texttt{cdef} for C-level variable declarations, typed \texttt{np.ndarray[np.float64\_t, ndim=1]} for buffer access, and \texttt{cimport} to pull \texttt{fabs} and \texttt{sqrt} from \texttt{libc.math}. Compiler directives (\texttt{boundscheck=False}, \texttt{wraparound=False}, \texttt{cdivision=True}) strip away Python-level safety checks in the inner loops. The module builds with a standard \texttt{setup.py} using \texttt{cythonize}. On anomaly scoring, Cython gives a 15.3$\times$ speedup at 1\,M rows---slower than pure NumPy here because NumPy's \texttt{median} is already highly optimized in C under the hood, but Cython wins on operations that don't have a clean NumPy one-liner (like our rolling-std kernel).

\subsection{Numba}

Five functions in \texttt{numba\_ops.py} use \texttt{@njit(parallel=True, cache=True)} with \texttt{prange} loops. Numba scores 5.1$\times$ on anomaly detection and 3.5$\times$ on rolling mean relative to pandas. The numbers are lower than Cython because Numba has JIT compilation overhead on the first call, and our benchmark only does one pass per size. In a production setting where the function is called repeatedly, Numba's cached compilation would narrow the gap.

We also wrote a Numba CUDA kernel (\texttt{@cuda.jit}) for GPU-based anomaly scoring. The kernel assigns one thread per residual, with blocks of 256 threads, uses \texttt{cuda.grid(1)} for index computation, transfers data to the device with \texttt{cuda.to\_device()}, and reads results back with \texttt{.copy\_to\_host()}. On machines without an NVIDIA GPU the code falls back gracefully---it just logs a warning and uses the CPU path.

\subsection{CuPy}

For the vectorized GPU path, we use CuPy, which has a nearly identical API to NumPy. The rolling-mean GPU function uses \texttt{cp.cumsum} and \texttt{cp.where} to compute the sliding window in one pass without any Python-level loops. Early in development we had a version with a Python \texttt{for} loop iterating over CuPy arrays, which was actually \textit{slower} than NumPy because every array access triggered a GPU--CPU synchronization. Rewriting it as fully vectorized CuPy ops fixed the issue.

\subsection{Parallelism and Concurrency}

Per-site model training is CPU-bound, so we use \texttt{ProcessPoolExecutor} with \texttt{multiprocessing.get\_context("spawn")} (the \texttt{fork} context is unsafe on macOS and deprecated since Python 3.12). Chunk I/O is I/O-bound, so that uses \texttt{ThreadPoolExecutor} instead.

We also built a \texttt{ThreadedResultCollector} class that uses the lower-level primitives: a \texttt{Queue} for task dispatch, a \texttt{Lock} for thread-safe writes to the shared results dictionary, and daemon \texttt{Thread} workers that pull from the queue until they receive a \texttt{None} sentinel. This is the classic producer--consumer pattern.

Table~\ref{tab:parallel} and Fig.~\ref{fig:parallel} show parallel training times. The speedup tops out at 1.48$\times$ with 4 workers. This is expected---LightGBM itself is multi-threaded internally, so spawning 4 processes on a 4-core laptop means each process competes for the same cores. At 8 workers, things actually slow down a bit due to context switching.

\subsection{PySpark}

For completeness, we wrote a standalone Spark pipeline (\texttt{spark\_pipeline.py}) that reads the data into a Spark DataFrame, uses \texttt{Window} functions for lag and rolling features, and evaluates with \texttt{RegressionEvaluator}. This would matter if the dataset were too large for a single machine; at 20\,M rows, pandas is fine, but the code is there if someone wanted to scale to the full 53.6\,M.

\subsection{Memory}

Using \texttt{float32} instead of \texttt{float64} everywhere cuts memory roughly in half (15.3\,MB $\to$ 7.6\,MB for a 2\,M-row DataFrame in our benchmark). All 8 configuration dataclasses use \texttt{@dataclass(slots=True)} to avoid per-instance \texttt{\_\_dict\_\_} overhead. Combined with chunked I/O, peak memory for the full pipeline stays under 4\,GB.

\section{Engineering}
% ============================================================================

The codebase has 15 Python modules under \texttt{src/}, a Click CLI with 7 subcommands (\texttt{run}, \texttt{benchmark}, \texttt{parallel-benchmark}, \texttt{profile}, \texttt{spark}, \texttt{quality}, \texttt{eda}), YAML-based configuration parsed into typed dataclasses, and a Makefile that wraps all commands. We wrote 19 \texttt{pytest} tests covering leakage safety (lag and rolling features must not see the future), Numba correctness against a NumPy reference implementation (tested within floating-point tolerance), anomaly threshold behavior, temporal train/val split properties, and one end-to-end integration test on synthetic data. Dependencies are pinned in \texttt{requirements.txt} and the README documents every step from download to execution.

\section{Limitations and Future Work}
% ============================================================================

A few honest limitations. First, we ran the pipeline on $\sim$7\,M rows (4 chunks) rather than the full 20\,M for most experiments, because training LightGBM on the full set takes several minutes and we wanted fast iteration. The final numbers we report (82\% RMSE improvement, etc.) are on this 7\,M-row subset. Second, the parallel speedup is modest because LightGBM already parallelizes internally---a model that was purely single-threaded (like a from-scratch decision tree) would show more dramatic multi-worker gains. Third, we don't have a UI; the output is CSVs and PNGs, which is fine for a pipeline but not great for a demo.

For future work: (1)~run on the full 53.6\,M-row dataset with both training years; (2)~build a Streamlit dashboard so operators can interactively explore predictions and anomalies; (3)~try transfer learning across sites for buildings with limited data; (4)~deploy the model behind a REST API for hourly inference.
